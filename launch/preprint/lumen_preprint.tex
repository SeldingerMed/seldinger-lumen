\begin{abstract}
Endovascular robotics and embodied medical AI need simulation environments that expose
the actual control problem: a slender instrument threading a deformable, branching,
image-guided lumen under safety constraints. Lumen is an open Apache-2.0 simulator for
that setting. It combines tube-intrinsic contact, a deformable anisotropic vessel wall,
torsion, friction, flow/clot coupling, synthetic fluoroscopy, pixel labels, Gymnasium
environments, and a safety-scored benchmark in one public stack. The simulator is built
on NVIDIA Newton and Warp, so the same code path supports differentiable mechanics and
GPU-parallel rollouts. Lumen is designed as a research substrate for reinforcement
learning, computer vision, imitation learning, and dataset tooling: it can generate
procedural anatomy, render paired schematic and fluoroscopic views, validate replayable
case bundles, build dataloader indexes, split episodes without leakage, and materialize
strict tensor batches. This preprint describes the design principles, implementation
surface, benchmark semantics, and relationship to prior open endovascular simulation
work such as CathSim. The public repository is
\url{https://github.com/SeldingerMed/seldinger-lumen}; project documentation and demos
are at \url{https://seldingermed.github.io/seldinger-lumen/}.
\end{abstract}

\section{Motivation}

Endovascular procedures are a natural frontier for embodied AI. The task is precise,
spatially constrained, image-guided, and safety-critical: a device must be advanced and
rotated through a vascular lumen while maintaining wall safety, selecting branches, and
responding to contact, tortuosity, stenosis, clot, and flow. Robotic systems already
make remote and radiation-sparing catheter operation realistic, while reinforcement
learning and imitation learning offer a route to policies that can reason over image
feedback and contact-rich instrument dynamics \cite{jayender2008,chi2019,pore2023,robertshaw2024}.

The bottleneck is not merely the availability of a Gym API. The bottleneck is whether a
simulator can make clinically meaningful failures visible to learning algorithms. A
rigid tube with a target-reaching reward can produce an agent that reaches the goal by
scraping through the wall. A pixel renderer without segmentation and keypoints creates
vision data but not a complete CV training artifact. A mechanical solver without
replayable episodes is hard to use as a dataset generator. Lumen is built around the
opposite premise: the mechanics, labels, benchmark, and data artifacts should be part
of the same environment.

\section{Design goals}

Lumen is organized around five design goals.

\textbf{1. The lumen is a stateful physical object.} The wall is not just a collision
mesh. It is represented as a shared lumen field with anisotropic deformability and
contact response. This lets the same quantity define the contact boundary, wall
displacement, and safety metrics.

\textbf{2. Contact should be tube-intrinsic.} A centerline parameterization provides
local coordinates for projecting the device into the vessel. Contact is expressed in
terms of radial excess relative to the current lumen field and injected into the
Newton/Warp solve. This avoids treating the vessel as an external triangle mesh when
the control problem is fundamentally luminal.

\textbf{3. RL success must be safety-scored.} The benchmark reports raw target reach,
safe success, unsafe success, maximum wall penetration, and return. The leaderboard
ranks safe success before raw success, so wall-violating policies do not dominate by
reward hacking.

\textbf{4. Vision data should be first-class.} Lumen can emit synthetic fluoroscopy and
luminal observations with device masks, vessel masks, tip/base keypoints, node
locations, actions, clinical metrics, and calibration metadata.

\textbf{5. The artifact should be reusable.} The open core is Apache-2.0, provides
typed Python APIs, exposes CLI workflows, and writes replayable case bundles rather
than one-off screenshots.

\subsection{Mechanics}

The device is represented as a slender continuum instrument running inside a tubular
or branching asset. Lumen uses a Newton/Warp backend for rod dynamics and differentiable
simulation, following the broader trend toward programmable differentiable physics
systems for robot learning \cite{todorov2012,macklin2022,freeman2021,degrave2019}.
The wall model is inspired by anisotropic arterial mechanics, especially
Holzapfel--Gasser--Ogden families of constitutive models for collagen-reinforced
arterial tissue \cite{holzapfel2000,gasser2006}. The rod/contact design draws on
elastic rod simulation and constrained time integration work in graphics and robotics
\cite{bergou2008,spillmann2007,servin2006,macklin2019}.

\subsection{Observations}

Lumen exposes compact state observations for fast control experiments and image
observations for CV/RL pipelines. Synthetic fluoroscopy includes paired image and label
products: device/vessel masks, device node positions, and tip/base keypoints. This is
valuable because synthetic data is most useful when it carries dense ground truth that
is expensive or impossible to annotate at scale in real fluoroscopy
\cite{tobin2017,tremblay2018,nikolenko2021}.

\subsection{Data workflow}

The CLI is intended to make the first research loop short:
\texttt{lumen play} renders a rollout; \texttt{lumen benchmark} writes scorecards;
\texttt{lumen capture} writes replayable case bundles; \texttt{lumen validate}
checks assets, observations, labels, and sidecars; \texttt{lumen index} writes a JSONL
dataloader index; \texttt{lumen split-index} produces episode-grouped train/validation
/test splits; and \texttt{lumen materialize-batch} writes strict NPZ smoke-test
batches. This workflow turns simulation into a dataset interface rather than a demo
script.

\section{Benchmark semantics}

The canonical benchmark separates goal achievement from safe goal achievement. Each
task reports:
\begin{itemize}
  \item \textbf{success rate}: the tip reached the target;
  \item \textbf{safe success rate}: the target was reached without crossing the
  wall-safety threshold;
  \item \textbf{unsafe success rate}: the target was reached only with a safety breach;
  \item \textbf{maximum penetration}: worst wall-safety violation observed;
  \item \textbf{return}: task reward after safety and efficiency terms.
\end{itemize}
The leaderboard ranks safe success first, then raw success, then lower maximum
penetration, then return. This is a deliberately clinical ordering: an agent that
arrives by perforating the wall has not solved the problem.

\section{Positioning relative to CathSim and prior simulators}

CathSim is the strongest open reference point for endovascular RL environments: it
packages a MuJoCo/dm\_control catheter environment, force sensing, and baseline agents
for autonomous catheterization research \cite{jianu2023,jianu2024}. Lumen builds on the
research direction CathSim helped establish, while targeting a broader substrate for
deformable mechanics and multimodal data generation.

Prior work in autonomous endovascular navigation has explored robotic catheterization,
reinforcement learning, image-guided control, and simulation-to-real transfer
\cite{jayender2008,chi2019,behr2019,karstensen2024,yao2025}. Lumen's contribution is
to expose a compact open stack where those experiments can share the same mechanics,
sensor outputs, and data contracts.

\section{Implementation surface}

The public repository provides:
\begin{itemize}
  \item procedural straight, stenotic, tortuous, and branching assets;
  \item Newton/Warp-backed guidewire simulation;
  \item Gymnasium navigation environments;
  \item synthetic fluoroscopy and luminal cameras;
  \item replayable case bundles with masks/keypoints;
  \item benchmark scorecards and leaderboards;
  \item validation and dataloader-index CLIs;
  \item typed Python package metadata and an Apache-2.0 license.
\end{itemize}

The quick-start installation path is:
\begin{verbatim}
git clone https://github.com/SeldingerMed/seldinger-lumen
cd seldinger-lumen
pip install -e ".[dev]"
lumen doctor
lumen play stenotic --out lumen-run
lumen benchmark lumen-bench
\end{verbatim}

\section{Conclusion}

Lumen is a launch point for open endovascular AI research: a differentiable,
GPU-parallel simulator that treats wall mechanics, contact, image formation, labels,
and safety-scored benchmarks as one integrated problem. It is designed for researchers
who want to train agents, generate CV data, test policies, and compare methods without
starting from a rigid-pipe toy task or a closed simulation stack. The open repository
and demos are available at \url{https://github.com/SeldingerMed/seldinger-lumen} and
\url{https://seldingermed.github.io/seldinger-lumen/}.
